## Description

The contract helpers parse individual fields from a peer-supplied redeem script without first proving that the script is the exact HTLC template generated by `create_contract_redeemscript`. `read_hashvalue_from_contract` only checks that instructions 3 and 4 look like `OP_HASH160 <20-byte hash>`, and the other readers similarly depend on fixed positions/offsets. In the legacy taker verification path, a malicious maker's `SenderContractTxInfo` is accepted by checking the transaction pays to the supplied redeem script, then reading hashvalue, locktime, and hashlock pubkey from that same script. A maker can therefore supply a script that carries the expected fields but changes later opcodes, e.g. replacing `OP_CHECKSIG` with `OP_FALSE`, so the victim accepts and signs/broadcast-tracks a contract that is not spendable through the intended hashlock/timelock branches. This can break recovery and deny the negotiated payout. I checked prior findings with `contract redeemscript template validation read_hashvalue_from_contract read_hashlock_pubkey_from_contract` and `malformed contract redeemscript accepted invalid HTLC template`; neither matched.
